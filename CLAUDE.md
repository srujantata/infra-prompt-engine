# infra-prompt-engine — LLM Context

## What This Repo Does
Python service that converts natural language prompts into Terraform HCL and automatically opens GitHub pull requests. Integrates Claude API (Anthropic) for code generation.

## Flow
```
User prompt (string)
  → Claude API (claude-sonnet, system prompt with Terraform rules)
  → JSON response: {main_tf, variables_tf, outputs_tf}
  → GitHub API: create branch → commit 3 files → open PR
  → GitHub Actions (in aws-eks-platform): terraform plan → comment on PR
  → Human reviews → merges → terraform apply
```

## Directory Structure
```
prompt_engine/
├── generate.py          ← Core: takes prompt string, returns PR URL
├── templates/           ← System prompt templates per resource type
└── tests/               ← Unit tests for HCL generation

.github/workflows/
└── test.yml             ← Run pytest on PR

requirements.txt
.env.example             ← ANTHROPIC_API_KEY, GITHUB_TOKEN, GITHUB_REPO
```

## Key Implementation Details
- **Model**: `claude-sonnet-4-6` (or latest Sonnet) — balance of speed + quality
- **System prompt rules**: Use terraform-aws-modules, always tag resources, gp3 storage, encryption on all resources, output 3 files as JSON
- **GitHub integration**: PyGithub library (`pip install PyGithub`)
- **Branch naming**: `claude/infra-YYYYMMDD-HHMMSS`
- **PR body**: includes original prompt, environment, "Generated by Claude" footer

## Usage
```bash
pip install -r requirements.txt
cp .env.example .env
# Fill in .env
python prompt_engine/generate.py "Create VPC with 3 AZs and private subnets"
# Returns: https://github.com/srujantata/aws-eks-platform/pull/N
```

## What NOT to Do
- Never hardcode API keys — always use env vars / .env (gitignored)
- Never skip `terraform validate` in the GitHub Actions plan step
- Never auto-apply without human PR review — always require merge gate
- Don't use `claude-opus` for bulk generation — too slow and expensive for IaC
